## Global Resources (Super Admin Panel only)

### Jugendherberge Resource

The Jugendherberge is a global model. Only `super_admin` can create, edit or delete venues and their rooms. Tenant users read Jugendherbergen when creating contracts ‚Äî they cannot modify them.

Filament resource lives in the Super Admin Panel, not the Camp Panel.

**Jugendherberge form fields:**
- Name, address, city, phone, email, website, notes

**JugendherbergeRoom relation manager (within Jugendherberge):**
- Name, floor, capacity
- Rooms are fixed per venue. Changes are rare (new building construction).
- No gender field ‚Äî gender-based bed assignment is handled via camp gender policy, not per-room.

### HostelContract ‚Äî Infolist Section on Camp View

Created by the Camp Manager after price negotiation with the JH. One contract per camp (`HasOne` from Camp).

**Not a RelationManager.** Shown as a Section inside the Camp infolist (ViewCamp). The section header has three conditional actions:
- **Add Contract** ‚Äî visible when no contract exists, opens a create modal
- **Edit** ‚Äî visible when contract exists, pre-fills the modal with current values
- **Delete** ‚Äî visible when contract exists, requires confirmation

**Form fields:**
- Hostel (searchable select from global `hostels` table)
- Price per person per night (decimal, EUR)
- Contracted participants (integer ‚Äî JH's agreed max)
- Contracted supporters (integer)
- Contract date (date, optional)
- Notes (cancellation terms, special conditions)

A camp without a contract can still be created. The cost calculator will be incomplete without it.

The FK is on `hostel_contracts.camp_id` (not on `camps`). `Camp::hostelContract()` is a `HasOne`.

### Cost Calculator Widget

A summary widget displayed on the Camp detail page. Shows the cost breakdown and calculates price per participant.

**Note: negative totals are valid and expected.** When `price = 0` (e.g. Vereins camp covered by donations), the total will show a deficit. This is correct behaviour ‚Äî do not add validation preventing negative values.

**Displays:**
- Accommodation cost: `contract.price_per_person_per_night √ó (predicted_participants + predicted_supporters) √ó camp_nights`
- Total expenses per category (summed from CampExpense records)
- Grand total
- **Predicted price per participant:** `(accommodation_cost + total_expenses) / predicted_participants`
- **Current confirmed price per participant:** `(accommodation_cost + total_expenses) / confirmed_registrations_count`

Both predicted and confirmed prices are shown separately. They diverge as registrations come in. This is intentional ‚Äî do not merge them.

Camp nights = `ends_at - starts_at` in days.

## Public Registration Form

Route: `/{tenant:slug}/camps/{camp}/register`
Plain Blade view. No login required. Tenant name shown at top.

The form structure is driven by `camp.target_group`. There are three distinct modes.

---

### Form Mode: `children`

The visitor is always a parent or guardian registering one or more children. No "myself / my child" question ‚Äî it is always a child.

**Visitor fields (once):**
- Guardian name
- Email (all notifications sent here)
- Phone (optional)

**Participant fields (repeater ‚Äî one or more children):**
- Child's full name
- Date of birth
- Gender (male / female)
- Allergies (optional)
- Medications (optional)
- Medical notes (optional ‚Äî label: "Anything else we should know")
- Emergency contact name
- Emergency contact phone

**Terms acceptance (required, once)**

`is_self = false` always. Each child in the repeater becomes its own `Participant` and its own `CampRegistration`.

---

### Form Mode: `adults`

The visitor registers only themselves. No child fields. Single participant per submission.

**Fields:**
- Full name
- Date of birth
- Gender (male / female)
- Email
- Phone (optional)
- Allergies (optional)
- Medications (optional)
- Medical notes (optional)
- Emergency contact name
- Emergency contact phone
- Terms acceptance (required)

`is_self = true` always. One submission produces exactly one `Participant` and one `CampRegistration`.

---

### Form Mode: `mixed`

Show a "Registering for" radio: **myself** or **my child**.

If **myself** ‚Üí render the adults form. `is_self = true`. Single participant.

If **my child** ‚Üí render the children form with repeater. `is_self = false`. Multiple participants possible.

---

### On Submission (all modes)

Run the following for each participant in the submission:
1. Find or create `Visitor` by email
2. Create `Participant` with health fields and correct `is_self` value
3. Count confirmed + pending registrations for this camp
4. If count < `camp.capacity` ‚Üí status = `pending`, `registered_at` = now
5. If count ‚â• `camp.capacity` ‚Üí status = `waitlisted`, assign `waitlist_position`, `registered_at` = now
6. Create `CampRegistration`
7. Send appropriate notification email

Steps 3‚Äì7 run once per participant, not once per submission. Each child gets its own independent registration, waitlist position, payment status, room assignment and notification emails.

## Waitlist Logic

**Registering:** `registered_at` is always set at form submission. `waitlist_position` assigned as current max + 1.

**Promotion trigger:** when a confirmed or pending registration is cancelled.

**Promotion process:**
1. Find lowest `waitlist_position` registration for this camp
2. Set status ‚Üí `pending`, clear `waitlist_position`
3. Send waitlist promotion email
4. Reorder remaining waitlisted registrations (renumber from 1)

Promotion is NOT automatic confirmation. Camp Manager still manually confirms and marks payment.

## Bed Assignment

The Camp Manager assigns beds in Filament after registration is confirmed.

- Rooms come from `jugendherberge_rooms` linked via the JH contract
- Filter rooms by gender policy of the camp
- Show bed count per room: X of Y assigned
- Assignment stored in `camp_room_assignments`

## Email Notifications

All emails queued. All include tenant name in header. All logged to `camp_notification_logs`.

| type | trigger | key content |
|---|---|---|
| `registration_received` | form submission, capacity available | IBAN, payment instructions, "not confirmed until payment verified" |
| `waitlisted` | form submission, camp full | waitlist position, "we will notify you" |
| `waitlist_promoted` | registration cancelled by another | IBAN, payment instructions, deadline warning |
| `confirmed` | Camp Manager confirms in Filament | camp dates, location, room if assigned |
| `payment_reminder` | manual button in Filament | outstanding amount, IBAN |
| `room_assigned` | room assignment saved | room name, camp dates |
| `cancelled` | cancellation action | brief message |
